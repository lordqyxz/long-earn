# ADR-005: 事件驱动回测框架替代向量化引擎

## 状态
Proposed

## 上下文
当前的向量化回测引擎（基于 Pandas MultiIndex 和 AST 表达式）在处理简单因子策略时速度极快，但在以下方面存在严重局限：
1. **可信性挑战**：虽然使用了 AST 白名单，但在处理复杂窗口函数或自定义逻辑时，极易引入“未来函数”（Look-ahead bias）。
2. **表达力不足**：无法实现状态机逻辑（例如：只有在满足 A 条件后，才在满足 B 条件时下单），难以支持复杂的止损、追踪止盈等动态风控。
3. **执行模拟粗糙**：交易成本通过权重 $\Delta W$ 简单扣除，无法模拟真实的订单簿状态、部分成交或复杂的执行算法。

## 决策
我们将从向量化架构转向 **事件驱动 (Event-Driven)** 架构。

### 核心优先级
1. **可信性 (Financial Fidelity)**：绝对禁止策略在 $T$ 时刻访问 $T$ 时刻及以后的任何数据。通过严格的事件流控制时间线。
2. **表达力 (Expressiveness)**：允许策略持有内部状态，支持基于事件触发的复杂逻辑，能够精准控制订单生命周期。
3. **速度 (Performance)**：作为次要目标。接受单步迭代带来的性能下降，通过优化 Python 循环（如使用 `numba` 或 `cython`）或部分向量化（在单个时间步内）来缓解。

## 架构方案
- **Event Loop**: 核心调度器，负责驱动时间前进并分发事件。
- **DataHandler**: 逐日/逐点喂入数据，确保数据可见性。
- **Strategy**: 状态化类，通过 `on_bar` / `on_event` 响应并产生 `Signal`。
- **Portfolio**: 管理持仓、资金，将 `Signal` 转换为 `Order`。
- **Broker**: 模拟撮合，将 `Order` 转换为 `Fill`（含滑点、佣金计算）。

## 影响
- **兼容性**：原有的 YAML DSL 需要升级以支持状态化描述，或引入轻量级 Python 策略定义。
- **开发量**：需要重新实现核心执行链路。
- **测试**：需要建立一套基于事件序列的回归测试集，验证 Net Return 的准确性。
